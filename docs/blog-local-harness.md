# 影视创作工具平台的 AI Coding Harness 实践：从上下文工程到生产闭环

> 本文不是介绍一个工具怎么使用，而是复盘我们在影视创作工具平台里，如何把 AI Coding 从“个人提效工具”推进到“团队可复用的生产级研发流程”。文中涉及的业务名、路径、字段和截图信息均已脱敏。

## 一、为什么要重新搭一套 Harness

影视创作工具平台的业务形态很适合 AI Coding，也很容易把 AI Coding 用坏。

一方面，平台里有大量相似但不完全相同的业务链路：剧集、分镜、资产、视频转码、内容生成、状态流转、异步任务、Kafka 消费、回调更新。AI 很擅长根据相似实现快速补齐代码。

另一方面，这类系统又天然有很多生产级约束：接口返回格式要统一，状态机不能错，异步任务要可追踪，数据库变更要兼容旧数据，线上问题需要能从 trace_id 一路查到日志、指标、Pod 状态和代码根因。

早期直接用 AI 对话写代码时，最常见的问题不是“写不出来”，而是“写得不像这个系统里的代码”：上下文靠人工粘贴，规范靠口头提醒，测试靠事后补，线上问题靠人去日志平台和 IDE 之间来回切。任务一长，AI 容易遗忘前面的约束；多人协同时，每个人沉淀的提示词和经验也很难共享。

所以我们真正要搭的不是一个“更长的 Prompt”，而是一套能被团队共同使用的 Harness：

- 让 AI 在统一上下文里理解项目；
- 让需求从 0 到 1 有固定质量门禁；
- 让代码生成、审查、测试、修复分工明确；
- 让线上 trace_id 能一键进入排障链路；
- 让每次执行结果反过来改进 Harness 本身。

## 二、第一步不是写 Prompt，而是建设团队共享上下文

在生产级项目里，AI 生成质量的上限，很大程度取决于它拿到的上下文质量。

影视创作工具平台不是一个单文件应用。一个需求可能同时涉及 API、Service、DAO、Schema、模型字段、异步任务、Kafka 消费、前端查询、测试数据和线上状态。让每个工程师在对话里临时解释这些背景，不仅低效，而且不稳定。

因此我们先做的是 Code Wiki 和上下文管理。

Code Wiki 的定位不是给人看的项目文档，而是给 Agent 消费的工程地图。它会把项目结构整理成几类稳定知识：模块职责、接口契约、请求链路、数据流转、公共工具、错误处理、状态机、测试模式和架构禁区。比如“入口层只做请求解析”“Service 负责编排 DAO/Gateway”“状态字段新增必须考虑旧数据默认值”“异步任务必须带 trace_id”等规则，都应该成为 AI 默认知道的背景。

更重要的是，Wiki 不是一次性文档。它需要有鲜度机制：当相关代码发生变化时，能知道哪部分知识已经过期；当某个模块经历重构，能局部更新对应章节，而不是每次把整份文档重写一遍。

这样做以后，多人协同的体验会发生变化。工程师不再各自维护一份“私藏提示词”，而是围绕同一套项目上下文工作。新人启动一个需求时，不需要先问“这个项目有什么规矩”；Reviewer Agent 也不只是凭通用代码规范审查，而是能按平台自己的分层、状态机和接口契约判断是否应该阻断。

这也是我们理解的上下文工程：**不是把更多材料塞给 AI，而是把团队真实依赖的工程知识整理成可追踪、可更新、可复用的上下文底座。**

## 三、从 0 到 1 的需求实现：把质量门禁前移

在影视创作工具平台里，一个看似简单的状态字段变更，可能影响创建接口、内容查询、异步消费、历史数据兼容和测试断言。如果只让 AI “帮我加个字段”，它很容易完成局部修改，却漏掉后面的链路。

因此我们把需求实现拆成一条固定流水线：先做需求理解和变更摘要，再生成代码，再进入审查和测试，最后输出变更报告。实际使用时，流程里会有明确的人类确认卡点：AI 会先总结涉及接口、影响文件、数据库变更、向后兼容策略和风险等级，工程师确认后才进入编码。

截图里的真实使用过程就体现了这一点：在进入代码修改前，Agent 会把“涉及接口、影响文件、DB 变更、兼容性、风险等级”列成摘要，并在 Phase 之间等待确认。这个卡点看似降低自动化程度，实际上是在把返工成本最高的地方提前暴露。

代码生成完成后，生成者不会自己给自己打分，而是进入独立审查和测试：

- 代码审查关注架构分层、复杂度、重复逻辑、边界处理和平台规范；
- 安全审查关注密钥、权限、注入风险、敏感信息和不该进入本次变更的历史问题；
- 单元测试 Agent 会围绕本次变更生成验证用例和回归保护用例；
- Test Runner 只负责执行和给出结果，不参与改代码；
- Debugger 只修复失败项，不扩大范围做重构。

这套机制的关键，是把主观判断变成可执行的 VERDICT。PASS 才继续，FAIL 就进入修复或人工确认。截图里可以看到，单元测试生成后会再交给 test-runner 执行；最终报告里也会沉淀每个 Phase 的结果、改动文件、测试覆盖、DB 变更提醒和是否需要人工执行 DDL。

我们刻意保留这种“流水线报告”，因为它不仅是给当前工程师看的，也是给后续 Review、排障和复盘看的。AI Coding 不是生成代码结束，而是要留下可审计的交付痕迹。

## 四、让 AI 接入生产，但必须先定义安全边界

很多线上问题，AI 单看代码很难判断。比如接口 500、状态未更新、异步任务没回调、内容查询结果不一致，这些问题真正的证据往往在生产日志、K8s 事件、Prometheus 指标和数据库状态里。

过去的排障路径通常是：人拿到 trace_id，打开日志平台查日志，再去监控平台看指标，再查 Pod 状态，再回 IDE 里找代码。AI 只能看到人复制过来的片段，很难形成完整证据链。

我们的做法是把生产观测能力封装成只读 MCP，并把它接入 Debugger / Monitor Agent。用户只需要提供 trace_id、环境和问题描述，Agent 就能按顺序收集证据：

1. 用 trace_id 查询 Loki 日志，还原请求链路和异常堆栈；
2. 查询对应服务的 Pod、Deployment 和 K8s Events；
3. 拉取错误率、P95 延迟、CPU、内存等指标；
4. 必要时做数据库只读比对；
5. 结合代码定位根因，输出排障报告；
6. 经人类确认后，才进入最小化修复和回归验证。

![影视创作工具平台生产观测集成](../images/film-platform-prod-observability.png)

这里最重要的是安全边界。MCP 不是让 AI 拿到一把万能钥匙，而是把生产能力封装成白名单、只读、可审计的工具集合。K8s 只能查询，不能 exec、delete、apply；数据库只能做必要的只读查询；所有生产修复前必须先给出根因分析并等待确认。

这套链路也可以自然融入团队日常协作。比如在飞书群里，值班同学可以让运维 Agent 查询某个服务最近 30 分钟的错误率、Pod 重启情况或 K8s 事件；在研发流程里，工程师可以用 trace_id 直接启动线上问题排查。AI 不再只是写代码的助手，而是进入了“发现问题、定位问题、修复问题、沉淀经验”的生产闭环。

## 五、Trace 自进化：让 Harness 自己变得更好

一套 Harness 刚搭起来时，最大的问题不是功能不够，而是不知道哪里最该优化。

是需求设计阶段问得不够细？是生成 Agent 经常漏掉状态兼容？是 Reviewer 规则太松？是测试 Agent 数据构造不稳定？还是线上排障时缺少某个监控源？

如果这些问题只能靠人工翻聊天记录，很难持续改进。我们引入 Trace 自进化机制，把每次执行过程变成结构化数据：任务类型、阶段、Agent、VERDICT、重试次数、失败原因、涉及扩展点、测试结果和最终报告都会进入轨迹。

这些轨迹积累后，可以做三件事。

第一，聚类失败模式。比如发现“状态字段新增”类需求经常在测试阶段失败，说明上下文里需要补充状态机兼容规则，或者测试生成策略需要强化旧数据场景。

第二，归因到 Harness 组件。失败不是泛泛地归因于“AI 不稳定”，而是尽量定位到某个规则、某个扩展点、某类上下文缺失。

第三，把改进候选变成实验。比如给代码生成阶段增加一条“新增状态字段必须说明默认值和旧数据迁移策略”的规则，再用后续任务验证通过率是否提升。

这就是 AHE（Agentic Harness Engineering）的价值：让 Harness 从“手工调 Prompt”变成可观测、可归因、可实验的工程系统。每次失败都不只是失败，而是下一次规则改进的样本。

## 六、实践效果：预留一块用数据说话

这部分我们后续会补充更完整的数据口径，目前建议从三个维度评估，而不是只看“节省了多少编码时间”。

**代码生成质量**：关注一次生成后进入审查的 PASS 率、Critical 问题数量、跨层引用问题数量、重复代码和边界条件问题数量。

**测试与交付质量**：关注单元测试生成数量、测试首次通过率、回归保护用例数量、最终流水线通过率，以及人工介入次数。

**线上稳定性与排障效率**：关注 trace_id 到根因报告的耗时、线上问题修复平均耗时、重复问题是否通过 Wiki/AHE 规则沉淀减少复发。

后续可以补充一张指标表，类似：

| 需求从确认到首版代码耗时 | 待补充 | 待补充 | 待补充 |
| 代码审查 Critical 数 | 待补充 | 待补充 | 待补充 |
| 单元测试首次通过率 | 待补充 | 待补充 | 待补充 |
| trace_id 排障耗时 | 待补充 | 待补充 | 待补充 |
| 重复问题复发率 | 待补充 | 待补充 | 待补充 |

这里的关键是：AI Coding 的提效不应该只用“生成了多少行代码”衡量，而要看它是否真的提高了交付质量、降低了线上风险，并把团队经验沉淀成下一次可复用的能力。

## 总结

这次实践给我们的最大启发是：生产级 AI Coding 的核心不是模型本身，而是 Harness Engineering。

在影视创作工具平台里，AI 要写好代码，必须先理解上下文；要稳定交付，必须有质量门禁；要进入生产，必须有只读、安全、可审计的工具边界；要持续变好，必须记录 Trace 并反向改进规则。

所以我们最终搭建的不是一个“会写代码的机器人”，而是一套围绕团队工程流程运转的 AI Coding 体系：Code Wiki 管上下文，Command 管流程，Sub-agent 管职责，MCP 管生产证据，Trace/AHE 管持续进化。

当这些能力连起来之后，AI 才真正从个人助手变成团队研发基础设施。
